# GitHub Issues — Complete Beginner Guide

GitHub Issues are one of the most important tools in modern software engineering.

Issues help teams:

* track bugs
* manage tasks
* discuss features
* organize development

Think of Issues as:

# project task management system

---

# SIMPLE DEFINITION

A GitHub Issue is:

# a tracked discussion/task/problem inside a repository

Issues are used for:

* bug reports
* feature requests
* improvements
* questions
* planning

---

# REAL-WORLD ANALOGY

Imagine school project.

Team creates task board:

```text id="m4v8tb"
□ Design homepage
□ Fix calculator bug
□ Add login feature
```

GitHub Issues work similarly.

---

# WHY ISSUES EXIST

Without Issues:

* bugs get forgotten
* tasks become messy
* communication becomes chaotic

Issues organize software development.

---

# VISUAL IDEA

```text id="x7m2wc"
Repository
   ├── Code
   ├── Pull Requests
   └── Issues
```

Issues are part of project management.

# WHAT AN ISSUE CONTAINS

Usually:

* title
* description
* labels
* comments
* status
* assignee

---

# ISSUE TITLE

Short summary.

Good example:

```text id="v8x1tb"
Fix navbar overlap on mobile devices
```

Bad example:

```text id="w4m9qc"
bug
```

---

# ISSUE DESCRIPTION

Explains:

* problem
* expected behavior
* actual behavior
* reproduction steps

---

# EXAMPLE BUG ISSUE

Title:

```text id="y6x2tb"
Calculator crashes on division by zero
```

Description:

```text id="m3v7wc"
Steps:
1. Enter 5
2. Divide by 0
3. Program crashes

Expected:
Show error message.
```

---

# WHY GOOD ISSUES MATTER

Clear issues help developers:

* understand problems quickly
* reproduce bugs
* prioritize work

---

# LABELS

Labels categorize issues.

Examples:

| Label            | Meaning            |
| ---------------- | ------------------ |
| bug              | something broken   |
| enhancement      | improvement        |
| documentation    | docs-related       |
| good first issue | beginner-friendly  |
| help wanted      | needs contributors |

---

# VISUAL LABEL EXAMPLE

```text id="q8m1tb"
[bug] Fix login crash
[enhancement] Add themes
```

---

# ASSIGNEES

Issues can be assigned to developers.

Example:

```text id="u5m4qc"
Assigned to Alice
```

Meaning:
Alice responsible for fixing it.

---

# COMMENTS & DISCUSSION

Developers discuss inside issue.

Example:

```text id="r2x9tb"
"I found root cause."
"Working on fix."
```

This improves collaboration.

---

# OPEN vs CLOSED ISSUES

# Open Issue

Task/problem still active.

---

# Closed Issue

Task solved/completed.

---

# VISUAL WORKFLOW

```text id="y9m1wc"
Issue Created
      ↓
Developer Assigned
      ↓
Code Written
      ↓
Pull Request Opened
      ↓
Issue Closed
```

---

# ISSUES + PULL REQUESTS

Very important relationship.

Example:

```text id="x4m7tb"
Issue #12 → Fix navbar bug
```

PR description:

```text id="t8v3qc"
Fixes #12
```

After merge:
GitHub automatically closes issue.

---

# WHY THIS IS POWERFUL

Now project history becomes connected:

```text id="p1x6tb"
Problem → Discussion → Code Fix → Merge
```

Excellent organization.
